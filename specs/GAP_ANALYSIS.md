## Executive Summary

## 1. Missing Agents

- Parallel memory pattern (separate from task-oriented threads)
- `LoreEpisode` nodes linked via `[:EXPANDS_UPON]` relationships
- Cross-conversation saga continuity (CLI → Telegram → CLI)
- "Captain-Context" linking (stories follow the user, not threads)
- Integration with Scribe's midnight reflection for saga progress

**Why Missing Matters**: The Bard is what transforms Klabautermann from a tool into a companion. Without it, the "tidbit" system is just random text injection without narrative continuity.

### 1.2 The Purser (State Synchronization) - MEDIUM PRIORITY

**What it is**: Handles bidirectional delta-synchronization with external APIs (Gmail, Calendar, Tasks).

**Key Features**:
- Delta-link pattern (only sync changes, not full state)
- External ID tracking for deduplication
- Expiration handling for deleted external items
- Rate limit compliance
- The Sieve integration for email filtering

**Why Missing Matters**: Without the Purser, the system lacks a formal pattern for keeping the graph in sync with external services without API abuse.

### 1.3 The Quartermaster (Config Management) - DOCUMENTED PARTIALLY

**Status**: Mentioned in MEMORY.md but not fully specified.

**Missing Details**:
- Hot-reload mechanism with file hash checking
- Graph-stored prompts for self-optimization
- A/B testing capability for prompt variants
- Runtime model switching (Sonnet ↔ Haiku)

### 1.4 The Officer of the Watch (Proactive Alerts) - NOT DOCUMENTED

**What it is**: A background agent that monitors for conditions requiring proactive notification.

**Key Features**:

- Proactive morning briefings
- Deadline approaching warnings
- Anomaly detection (unusual patterns)
- Respects the "Rule of Silence" (only interrupt for SHIPWRECK-level events)

### 1.5 The Cartographer (Community Detection) - NOT DOCUMENTED

**What it is**: Identifies "Knowledge Islands"—clusters of highly related nodes representing major life themes.

**Key Features**:
- Louvain/Leiden community detection via Neo4j GDS
- Community nodes with `[:PART_OF_ISLAND]` relationships
- High-level "Island Summaries" generated by the Scribe
- Multi-level retrieval: Macro (Islands) → Meso (Projects) → Micro (Entities)

**Graph Structure**:
```cypher
// Community Node
CREATE (c:Community {
    uuid: randomUUID(),
    name: 'Work Life',
    summary: 'Professional activities and career...',
    node_count: 247,
    detected_at: timestamp()
})

// Link entities to communities
MATCH (p:Project {name: 'Q1 Budget'})
CREATE (p)-[:PART_OF_ISLAND]->(c)
```

---

### 1.6 The Hull Cleaner (Graph Pruning) - NOT DOCUMENTED

**What it is**: Removes "barnacles"—weak relationships and redundant nodes that degrade performance.

**Key Features**:
- Prune relationships with weight < 0.2 after 90 days
- Transitive reduction (remove redundant paths)
- Message node deletion after successful summarization
- Self-correcting "hallucination debt"

## 2. Ontology Gaps: Personal Life

### 2.1 Current State (Work-Focused)

The current ONTOLOGY.md is heavily biased toward professional contexts:
- Person, Organization, Project, Goal, Task
- WORKS_AT, CONTRIBUTES_TO, REPORTS_TO

### 2.2 Missing Personal Life Entities

| Entity | Description | Properties |
|--------|-------------|------------|
| **FamilyMember** | Extended Person with family role | `relationship_type`, `birthday`, `anniversary` |
| **Friend** | Social connection outside work | `met_how`, `shared_interests` |
| **Hobby** | Leisure activities | `name`, `frequency`, `skill_level` |
| **HealthMetric** | Wellness tracking | `type`, `value`, `unit`, `recorded_at` |
| **Pet** | Animal companions | `name`, `species`, `breed`, `birthday` |
| **Milestone** | Personal achievements | `name`, `achieved_at`, `significance` |
| **Routine** | Recurring personal activities | `name`, `frequency`, `time_of_day` |
| **Preference** | Personal likes/dislikes | `category`, `item`, `sentiment` |

### 2.3 Missing Personal Relationships

```cypher
// Family relationships
(Person)-[:FAMILY_OF {role: 'spouse|parent|child|sibling'}]->(Person)
(Person)-[:SPOUSE_OF]->(Person)
(Person)-[:PARENT_OF]->(Person)
(Person)-[:CHILD_OF]->(Person)
(Person)-[:SIBLING_OF]->(Person)

// Social relationships
(Person)-[:FRIEND_OF {since: timestamp}]->(Person)
(Person)-[:KNOWS {context: 'gym|church|school'}]->(Person)

// Interest relationships
(Person)-[:PRACTICES]->(Hobby)
(Person)-[:INTERESTED_IN]->(Topic)
(Person)-[:PREFERS {sentiment: 'likes|dislikes'}]->(Preference)

// Pet relationships
(Person)-[:OWNS]->(Pet)
(Person)-[:CARES_FOR]->(Pet)

// Health tracking
(Person)-[:RECORDED]->(HealthMetric)
(HealthMetric)-[:MEASURED_ON]->(Day)
```

### 2.4 Knowledge Islands for Life Themes

The Consumer App section of IDEA.md describes a visual "Locker" with Islands:
- **Work Island**: Professional activities
- **Family Island**: Family matters
- **Hobbies Island**: Leisure activities
- **Health Island**: Wellness tracking
- **Social Island**: Friendships and community

These should be first-class Community nodes in the ontology.

### 3.2 What's Missing

#### The Parallel Memory Pattern

The Bard maintains its own memory space separate from task threads:

```python
# Query for Bard context (separate from Researcher)
MATCH (p:Person {uuid: $captain_uuid})<-[:TOLD_TO]-(le:LoreEpisode)
WHERE le.saga_id = $current_saga
RETURN le ORDER BY le.chapter DESC LIMIT 3
```

#### Cross-Conversation Saga Continuity

Stories should persist across:
- Different threads
- Different channels (CLI, Telegram)
- Different sessions

```cypher
// LoreEpisode Node
CREATE (le:LoreEpisode {
    uuid: randomUUID(),
    saga_id: 'the-great-maelstrom',
    chapter: 3,
    content: 'The fog thickened as we approached...',
    told_at: timestamp()
})

// Link to previous chapter
MATCH (prev:LoreEpisode {saga_id: 'the-great-maelstrom', chapter: 2})
CREATE (le)-[:EXPANDS_UPON]->(prev)

// Link to Captain (not Thread)
MATCH (p:Person {uuid: $captain_uuid})
CREATE (le)-[:TOLD_TO]->(p)
```

#### Saga Management


## 4. Additional Missing Features

### 4.1 The Sieve (Email Filtering)

A multi-tiered email filtering pipeline that classifies incoming emails:
- **Boarding Party Detection**: Prompt injection patterns
- **Noise Filtering**: Newsletters, promotions, transactional
- **Knowledge Value Check**: Minimum content threshold
- **Whitelist Support**: VIP senders bypass filtering

### 4.2 Deep Salvage Pipeline

Automatic entity extraction from files:
- PDF parsing with summarization
- Image analysis via vision models
- Automatic linking to Projects and Persons
- "Disputed Fact" flagging for contradictions

### 4.3 Multi-Level Retrieval

The Researcher should support "zoom" mechanics:
- **Macro View**: Query Community/Island nodes
- **Meso View**: Query Note and Project nodes
- **Micro View**: Query Entity and Episode nodes

### 4.4 The Bridge Dashboard

A React-based web UI with:
- The Chronometer (System Health)
- The Captain's Log (Live log streams)
- The Horizon (Chat interface)
- The Locker (Graph explorer)
- The Scribe's Ledger (Journal timeline)
- Storm Warnings panel

### 4.5 Barnacle Scraping & Pruning

Automatic graph maintenance:
- Weak relationship pruning (weight < 0.2, age > 90 days)
- Transitive reduction
- Message node cleanup post-summarization
- Duplicate entity merging

### 4.6 Self-Optimizing Prompts

Store prompts in the graph for:
- A/B testing prompt variants
- User-driven personality adjustments ("tone it down")
- Automatic retirement of underperforming prompts
