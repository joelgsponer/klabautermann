# OAuth Flows - Purser Devnotes

## Google OAuth2 Setup

### Bootstrap Script (T027)

**Purpose**: One-time interactive setup to obtain refresh token for headless operation.

**Script**: `scripts/bootstrap_auth.py`

**Flow**:
1. Check prerequisites (credentials.json exists)
2. Open browser for user authorization (localhost:8080 callback)
3. Exchange authorization code for tokens
4. Verify credentials with test API calls
5. Extract client_id and client_secret from credentials.json
6. Store all three values in .env
7. Create timestamped backup of .env

**Required Scopes**:
- `https://www.googleapis.com/auth/gmail.modify` - Read, send, draft emails
- `https://www.googleapis.com/auth/calendar.events` - Read, create calendar events

**Critical Flags**:
- `prompt="consent"` - Force consent screen to ensure refresh token is issued
- `access_type="offline"` - Required to receive refresh token

### Prerequisites

1. **Google Cloud Project**:
   - Create at https://console.cloud.google.com
   - Enable Gmail API and Calendar API
   - Create OAuth2 credentials (Desktop application type)
   - Download credentials.json to project root

2. **Environment Variables** (in .env):
   - `GOOGLE_CLIENT_ID` - From credentials.json
   - `GOOGLE_CLIENT_SECRET` - From credentials.json
   - `GOOGLE_REFRESH_TOKEN` - Generated by bootstrap script

### Troubleshooting

#### "No refresh token received"

**Cause**: App was already authorized without revoking previous consent.

**Fix**:
1. Go to https://myaccount.google.com/permissions
2. Find your app and revoke access
3. Run `python scripts/bootstrap_auth.py` again

#### "Credential verification failed"

**Cause**: APIs not enabled or invalid credentials.

**Fix**:
1. Verify Gmail API and Calendar API are enabled in Google Cloud Console
2. Check credentials.json is valid (not expired, correct project)
3. Ensure OAuth consent screen is configured

#### "Rate limit exceeded"

**Cause**: Too many API requests during testing.

**Fix**:
- Wait 60 seconds and try again
- Check Google Cloud Console quotas
- Implement rate limiting in production code (see T028)

### Token Lifecycle

1. **Initial Authorization**: User grants consent via browser
2. **Refresh Token**: Long-lived (until revoked), stored in .env
3. **Access Token**: Short-lived (1 hour), never stored, refreshed on-demand
4. **Token Refresh**: Automatic via google-auth library when access token expires

### Security Best Practices

1. **Never commit credentials**:
   - .env (contains refresh token)
   - credentials.json (contains client secret)
   - .google_token.json (temporary token file)
   - .env.backup.* (timestamped backups)

2. **Minimal scopes**: Only request what's needed
   - gmail.modify (not gmail.readonly for full access)
   - calendar.events (not calendar for full access)

3. **Token rotation**: Refresh tokens can be revoked and regenerated
   - User revokes at https://myaccount.google.com/permissions
   - Run bootstrap script again to obtain new token

4. **Environment isolation**: Separate credentials per environment
   - Development: Personal Google account
   - Production: Service account or dedicated user
