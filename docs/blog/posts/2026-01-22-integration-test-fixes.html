Mocking the Right Things

Tests tell stories. Sometimes those stories are "this code works." Sometimes they're "your mocks are stale and the real code moved on without you."

Today was the second kind.

The Symptoms

Four integration test issues sat open: #247 (Archivist to Thread), #248 (Scribe to Day nodes), #249 (Skills to Executor), #250 (Context to Response). All P1-high priority. All claiming to test critical integration flows.

All failing with the same theme: mocks returning the wrong shape.

KeyError: 'link_count'

AttributeError: Executor does not have the attribute '_handle_email_action'

The Root Causes

Problem 1: Missing link_count

The Archivist creates Note nodes when archiving threads. The create_note_with_links function returns both the note UUID and how many entities were linked. The mock only returned the UUID.

# What the mock returned
mock_neo4j_client.execute_write = AsyncMock(
    return_value=[{"uuid": str(uuid.uuid4())}]
)

# What the code expected
link_count: int = int(result[0]["link_count"]) if result else 0

The fix: mock the right function at the right level.

with (
    patch("klabautermann.agents.archivist.summarize_thread", new_callable=AsyncMock),
    patch("klabautermann.agents.archivist.create_note_with_links", new_callable=AsyncMock) as mock_create_note,
):
    mock_create_note.return_value = {
        "note_uuid": expected_note_uuid,
        "entity_link_count": 1,
    }

Problem 2: Renamed Methods

The test tried to mock _handle_email_action. The Executor has _handle_gmail_send.

Code evolves. Tests that mock internal methods need to evolve with them.

# Before (wrong)
with patch.object(executor, "_handle_email_action", new_callable=AsyncMock):

# After (right)
with patch.object(executor, "_handle_gmail_send", new_callable=AsyncMock):

Problem 3: Wrong Import Path

The orchestrator v2 test patched klabautermann.agents.orchestrator.get_recent_summaries. That module doesn't export that function. It's imported from klabautermann.memory.context_queries into _orchestrator.py.

When mocking, patch where the function is used, not where it's defined.

# Before (wrong)
patch("klabautermann.agents.orchestrator.get_recent_summaries", ...)

# After (right)
patch("klabautermann.agents.orchestrator._orchestrator.get_recent_summaries",
      new_callable=AsyncMock, ...)

The Philosophy

Tests define what code should do. If tests fail, fix the code.

But what if tests fail because mocks are wrong?

Then you fix the mocks. The tests were right about what behavior to verify. They were just speaking a language the code no longer understood.

The Results

  • 35 tests pass in test_memory_lifecycle.py and test_orchestrator_v2.py
  • 4 issues closed: #247, #248, #249, #250
  • Integration test coverage for Archivist, Scribe, Executor, and Context flows validated

The Lesson

Mocks are contracts. They say "when you call this, you get that." When the real code changes what "this" looks like or what "that" contains, the contract is broken.

The tests didn't lie. They told us exactly where the contracts had drifted.

The voyage continues. With better mocks.
