Guarding the Schema: Ontology Validation and Pre-Extraction

Graphiti handles entity extraction well, but what happens when the LLM hallucinates a relationship type that doesn't exist in our ontology? Or creates an entity type we don't support?

Previously: silent corruption. Now: validation catches it before it reaches the graph.

The Problem

Our ontology defines strict types. A Person can WORKS_AT an Organization. A Task can BLOCKS another Task. But nothing stops an LLM from inventing EMPLOYED_BY or creating a Company instead of Organization.

These inconsistencies pollute the graph. Queries fail to find expected relationships. Community detection groups unrelated nodes. The knowledge graph becomes a knowledge swamp.

The Solution: Validation Layer

The new OntologyValidator validates every extracted entity and relationship before ingestion:

from klabautermann.core import (
    OntologyValidator,
    ExtractedEntity,
    ExtractedRelationship,
    ExtractionResult,
)

validator = OntologyValidator(strict=False)

# Validate an entity
entity = ExtractedEntity(
    name="John Smith",
    entity_type="Person",  # Valid - in NodeLabel enum
    properties={"email": "john@example.com"},
)
result = validator.validate_entity(entity)
print(result.is_valid)  # True

# Invalid entity type
entity = ExtractedEntity(
    name="Acme",
    entity_type="Company",  # Invalid - should be Organization
)
result = validator.validate_entity(entity)
print(result.is_valid)  # False
print(result.issues[0].code)  # INVALID_ENTITY_TYPE

What Gets Validated

The validator performs five types of checks:

  1. Entity type validation: Must be in NodeLabel enum (Person, Organization, Task, etc.)
  2. Relationship type validation: Must be in RelationType enum (WORKS_AT, REPORTS_TO, etc.)
  3. Constraint validation: Source and target types must be compatible:
    • WORKS_AT: Person -> Organization (valid)
    • WORKS_AT: Organization -> Person (invalid)
  4. Property type validation: Email should be string, not int
  5. Enum value validation: Task status must be "todo", "in_progress", "done", or "cancelled"

Severity Levels

Not all issues are created equal:

class ValidationSeverity(str, Enum):
    ERROR = "error"    # Blocks ingestion
    WARNING = "warning"  # Logged, allows ingestion
    INFO = "info"      # Informational only

ERROR: Invalid entity type, invalid relationship type, constraint violations. These block ingestion.

WARNING: Wrong property type (email as int), invalid enum value. Logged but allowed.

INFO: Low confidence extraction (<50%). Worth knowing, not worth blocking.

Strict Mode

For production environments, enable strict mode:

validator = OntologyValidator(strict=True)

# Now warnings become errors
entity = ExtractedEntity(
    name="My Task",
    entity_type="Task",
    properties={"status": "pending"},  # Should be "todo"
)
result = validator.validate_entity(entity)
print(result.is_valid)  # False in strict mode, True otherwise

LLM Pre-Extraction

Validation is reactive - it catches bad data after Graphiti extracts it. Pre-extraction is proactive - it uses a fast model (Haiku) to extract entities before Graphiti, giving us early validation.

from klabautermann.agents import (
    PreExtractionEngine,
    PreExtractionConfig,
)

config = PreExtractionConfig(
    enabled=True,
    model="claude-3-5-haiku-latest",  # Fast
    min_confidence=0.5,  # Filter low-quality extractions
    validate_ontology=True,
)

engine = PreExtractionEngine(anthropic_client, config)

result, validation = await engine.extract(
    "I met John Smith (john@example.com), PM at Acme Corp."
)

# Pre-extraction found these entities
for entity in result.entities:
    print(f"{entity.entity_type}: {entity.name} ({entity.confidence:.0%})")
# Person: John Smith (95%)
# Organization: Acme Corp (90%)

Ingestor Integration

The Ingestor agent now optionally runs pre-extraction before Graphiti:

from klabautermann.agents.ingestor import Ingestor
from klabautermann.agents import PreExtractionConfig

ingestor = Ingestor(
    graphiti_client=graphiti,
    neo4j_client=neo4j,
    anthropic_client=anthropic,  # Enables pre-extraction
    pre_extraction_config=PreExtractionConfig(
        min_confidence=0.5,
        strict_validation=False,
    ),
)

The flow is now:

  1. Clean input (remove role prefixes, roleplay markers)
  2. Pre-extract entities with Haiku (fast)
  3. Validate against ontology
  4. Log validation results for audit
  5. Pass to Graphiti for final extraction and storage

If pre-extraction fails, Graphiti still runs. It's a quality layer, not a gate.

The Prompt

Pre-extraction uses a structured prompt that teaches Haiku about our ontology:

VALID ENTITY TYPES (use exact casing):
Community, Day, Event, Goal, HealthMetric, Hobby, JournalEntry,
Location, LoreEpisode, Message, Milestone, Note, Organization,
Person, Pet, Preference, Project, Resource, Routine, Tag, Task, Thread

VALID RELATIONSHIP TYPES (use exact casing):
ACHIEVES, AFFILIATED_WITH, ASSIGNED_TO, ATTENDED, ATTACHED_TO,
BLOCKS, CARES_FOR, CHILD_OF, CONTAINS, CONTRIBUTES_TO, ...

EXTRACTION RULES:
1. Only extract entities that are explicitly mentioned
2. Assign confidence scores (0.0-1.0) based on how clearly the entity is identified
3. For relationships, both source and target entities must be extracted
4. Use WORKS_AT for employment (not "employed_by")
5. Names should be title-cased ("John Smith" not "john smith")
...

The LLM returns structured JSON that maps directly to our Pydantic models.

Relationship Constraints

The validator knows which entity types can connect via which relationships:

RELATIONSHIP_CONSTRAINTS = {
    "WORKS_AT": ({"Person"}, {"Organization"}),
    "REPORTS_TO": ({"Person"}, {"Person"}),
    "PART_OF": ({"Task", "Project"}, {"Project"}),
    "BLOCKS": ({"Task"}, {"Task"}),
    "SPOUSE_OF": ({"Person"}, {"Person"}),
    "OWNS": ({"Person"}, {"Pet"}),
    # ...
}

If the LLM extracts Organization -[WORKS_AT]-> Person, validation fails with INVALID_SOURCE_FOR_RELATIONSHIP.
