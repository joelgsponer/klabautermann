All Good Tales Must End: Saga Lifecycle Management

The Bard of the Bilge has been telling tales since we introduced it. But like any good storyteller, the Bard needed to learn when a story is finished - and when it's time to let one fade into memory.

The Problem with Endless Sagas

Without lifecycle management, sagas could theoretically run forever. A story started in January might still be "active" in December, accumulating chapters that lost all narrative coherence. Or the Bard might start telling so many tales that none of them ever concluded.

We needed rules. Not the kind that kill creativity, but the kind that give stories proper endings.

Four Rules for the Bard

We implemented four lifecycle constraints:

1. Max Chapters Per Saga (#118)

Every saga concludes after 5 chapters. A satisfying narrative arc.

config = BardConfig(
    max_saga_chapters=5  # Default: 5
)

# When the Bard tries to continue a completed saga:
try:
    await bard._continue_saga(completed_saga)
except SagaCompleteError as e:
    # "Saga 'The Chronicle of the Haunted Hash' is complete with 5 chapters"
    pass

2. Max Active Sagas (#119)

No more than 3 sagas can be active at once. The Bard can't juggle too many tales.

config = BardConfig(
    max_active_sagas=3  # Default: 3
)

# When trying to start a fourth saga:
try:
    await bard.start_new_saga()
except SagaLimitReachedError as e:
    # "Maximum 3 active sagas reached. Active: Saga A, Saga B, Saga C"
    pass

3. Saga Timeout (#120)

Sagas auto-complete after 30 days of inactivity. No more zombie tales.

config = BardConfig(
    saga_timeout_days=30  # Default: 30
)

# The Bard can archive old sagas:
archived = await bard.archive_timed_out_sagas()

# Each gets a closing chapter:
# "And so the tale of The Legend of the Daemon fades into the mists
# of memory, waiting perhaps for another day to be told..."

4. Min Time Between Chapters (#121)

At least 1 hour must pass between chapters. Even legends need time to breathe.

config = BardConfig(
    min_chapter_interval_hours=1.0  # Default: 1.0
)

# When the Bard tries to add a chapter too quickly:
try:
    await bard._continue_saga(recent_saga)
except ChapterTooSoonError as e:
    # "Cannot add chapter to 'The Ballad' for 0.5 more hours"
    pass

Graceful Degradation

The key insight: these constraints shouldn't break the response. When the Bard can't continue a saga, it falls back to a standalone tidbit instead:

async def salt_response(self, clean_response, ...):
    if active_saga and should_continue:
        try:
            tidbit, chapter = await self._continue_saga(active_saga)
            return SaltResult(is_continuation=True, ...)
        except SagaLifecycleError:
            # Fall through to standalone tidbit
            pass

    # Standalone tidbit as fallback
    tidbit = self._generate_standalone_tidbit()
    return SaltResult(is_continuation=False, ...)

The user never sees an error. They just get a tidbit instead of a saga continuation.

The Exception Hierarchy

All saga errors inherit from SagaLifecycleError:

class SagaLifecycleError(Exception):
    """Base exception for saga lifecycle errors."""
    pass

class SagaCompleteError(SagaLifecycleError):
    """Saga reached max chapters."""
    pass

class SagaLimitReachedError(SagaLifecycleError):
    """Too many active sagas."""
    pass

class ChapterTooSoonError(SagaLifecycleError):
    """Not enough time since last chapter."""
    pass

class SagaTimedOutError(SagaLifecycleError):
    """Saga was inactive too long."""
    pass

This lets the salt_response method catch any lifecycle error with a single except SagaLifecycleError.

New Process Message Operations

Two new operations for saga management:

# Archive all timed-out sagas
msg = AgentMessage(
    payload={"operation": "archive_timed_out"},
    ...
)
response = await bard.process_message(msg)
# {"archived": [...], "count": 2}

# Get all active sagas with limit info
msg = AgentMessage(
    payload={"operation": "get_active_sagas"},
    ...
)
response = await bard.process_message(msg)
# {"active_sagas": [...], "count": 2, "max_active": 3}

The Timeout Check

When fetching an active saga, we now check if it's timed out:

async def _get_active_saga(self):
    timeout_ms = self.bard_config.saga_timeout_days * 24 * 60 * 60 * 1000
    cutoff_ms = now_ms - timeout_ms

    # Query returns saga data...

    saga = ActiveSaga(
        saga_id=row["saga_id"],
        saga_name=row["saga_name"],
        last_chapter=row["last_chapter"],
        last_told=row["last_told"],
        is_timed_out=row["last_told"] < cutoff_ms,  # New flag
    )
    return saga

The is_timed_out flag means we don't need a second query to check timeout status.
