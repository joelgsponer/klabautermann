The Bard Finds Its Voice

Every ship needs one voice, not a chorus. Today the Bard of the Bilge became the sole owner of Klabautermann's personality.

The Problem: Personality Scattered Across the Deck

Personality was everywhere and nowhere. The Orchestrator's SYSTEM_PROMPT contained nautical lexicon guidelines. Storm mode detection lived in the Orchestrator. The Bard only added tidbits 7% of the time.

Three problems:

  • Inconsistent voice - different prompts, different tones
  • Hard to maintain - change personality meant touching multiple files
  • Channel-agnostic - same output for CLI, Telegram, and API

The Solution: Single Owner

The Bard is now responsible for everything presentation:

# Before: Orchestrator handled personality
response = "Found Sarah in the database. She works at Acme."
# That's it. Functional. Sterile.

# After: Bard transforms with personality
result = await bard.apply_personality(
    clean_response="Found Sarah in the database. She works at Acme.",
    channel="telegram",
    trace_id="trace-123"
)

# "Scouted the horizon and found Sarah in The Locker.
#  She's sailing with Acme these days.
#  _Reminds me of the time I navigated the Great Maelstrom..."

LLM-Powered Voice

The Bard now uses Haiku for personality rewriting. Not just appending tidbits - actually transforming the response:

PERSONALITY_PROMPT = """You are the voice of Klabautermann...

PERSONALITY TRAITS:
- Efficient: Answer the question first, nautical color second
- Witty: Dry humor, never slapstick
- Knowledgeable: Quiet confidence
- Humble: Admit when you don't know

NAUTICAL LEXICON (use naturally, never forced):
- Database/memory → "The Locker"
- Search → "Scout the horizon"
- Task list → "The Manifest"
- Error → "Rough waters"

VOICE GUIDELINES:
- NEVER use "Arrr", "Matey", "Shiver me timbers"
- Nautical metaphors should feel natural
...

Rewrite this response with personality:
{response}"""

Temperature 0.8 for variety. Haiku for speed. Natural variety in every response.

Storm Mode: When to Shut Up

Sometimes the Captain needs facts, not flavor. Storm mode detection moved from Orchestrator to Bard:

STORM_KEYWORDS = ["urgent", "emergency", "critical", "asap", "immediately", "crisis"]

def _detect_storm_mode(self, text: str) -> bool:
    if not self.bard_config.storm_mode_enabled:
        return False
    text_lower = text.lower()
    return any(kw in text_lower for kw in self.bard_config.storm_keywords)

# Storm mode active:
# Input: "URGENT: Server down!"
# Output: "URGENT: Server down!"  (no personality, no tidbits)

When the seas are rough, the Bard stays silent.

Channel-Aware Formatting

Different channels need different formatting:

def format_for_channel(self, response: str, channel: str | None) -> str:
    if channel == "telegram":
        # Keep markdown formatting
        return response
    if channel == "cli":
        # Plain text
        return response
    if channel == "api":
        # Clean, no formatting
        return response
    return response

The foundation is laid. Future enhancements: ASCII art headers for CLI, proper Markdown escaping for Telegram.

The New Flow

User Input
    ↓
Orchestrator (clean functional response)
    ↓
Bard.apply_personality()
    ├─ 1. Auto-detect storm mode
    ├─ 2. LLM rewrite with Haiku (if not storm)
    ├─ 3. Add tidbits (7% chance)
    └─ 4. Format for channel
    ↓
Final response to user

PersonalityResult

The Bard returns a structured result:

@dataclass
class PersonalityResult:
    original_response: str
    final_response: str
    personality_applied: bool
    storm_mode: bool = False
    tidbit_added: bool = False
    tidbit: str | None = None
    saga_id: str | None = None
    channel: str | None = None

The Orchestrator knows exactly what happened. Logging is comprehensive. Debugging is straightforward.
