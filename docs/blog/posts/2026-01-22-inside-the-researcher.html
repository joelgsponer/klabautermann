Inside the Researcher

The Researcher is Klabautermann's Librarian. When you ask "Who did Sarah work for last year?" or "What's blocking my project?", it doesn't just grep through a database. It reasons about how to find the answer, executes multiple search strategies in parallel, and synthesizes findings into something useful.

This post goes deep into how it works.

The Four-Phase Pipeline

Every query flows through four phases:

  1. Plan - Claude Opus analyzes your query and generates a search plan
  2. Execute - Multiple search techniques run in parallel
  3. Aggregate - Results are deduplicated and scored
  4. Synthesize - Opus combines findings into a Graph Intelligence Report

The key insight: searching a knowledge graph isn't one technique - it's orchestrating several.

Phase 1: Search Planning with Opus

When your query arrives, the Researcher doesn't immediately search. It thinks first.

Claude Opus receives a structured prompt describing four available search techniques and the graph schema. It analyzes your query and outputs a SearchPlan:

{
  "original_query": "Who did Sarah work for last year?",
  "reasoning": "Temporal employment query. Need STRUCTURAL for WORKS_AT
               with historical filter. ENTITY_FULLTEXT to find Sarah's
               node. VECTOR as semantic fallback.",
  "strategies": [
    {
      "technique": "ENTITY_FULLTEXT",
      "query": "Sarah",
      "limit": 3,
      "rationale": "Find Sarah's entity node"
    },
    {
      "technique": "STRUCTURAL",
      "cypher_pattern": "WORKS_AT_HISTORICAL",
      "params": {"name_pattern": "(?i).*sarah.*"},
      "time_range": {"as_of": 1704067200},
      "rationale": "Historical employment as of Jan 2025"
    },
    {
      "technique": "VECTOR",
      "query": "Sarah employment job work",
      "rationale": "Semantic fallback"
    }
  ],
  "zoom_level": "micro"
}

The LLM is choosing search strategies based on semantic understanding - not keyword matching. "Last year" becomes a timestamp. "Work for" becomes a WORKS_AT relationship pattern. The query structure, not surface words, drives the plan.

Fallback When Planning Fails

If Opus times out or returns unparseable JSON, the Researcher doesn't crash. It falls back to a safe default: vector search plus entity fulltext. Conservative, but functional.

def _fallback_plan(self, query: str) -> SearchPlan:
    return SearchPlan(
        original_query=query,
        reasoning="Fallback: LLM planning unavailable",
        strategies=[
            SearchStrategy(
                technique=SearchTechnique.VECTOR,
                query=query,
                limit=10,
            ),
            SearchStrategy(
                technique=SearchTechnique.ENTITY_FULLTEXT,
                query=query,
                limit=5,
            ),
        ],
    )

Phase 2: Parallel Execution

Once planned, all strategies execute concurrently with asyncio.gather. This is where time gets saved - instead of serial queries, we fan out.

The Four Search Techniques

VECTOR - Semantic Similarity

Uses Graphiti's embedding search. Your query becomes a vector, and we find facts whose embeddings are geometrically close. Great for "remind me about..." or conceptual queries where you don't know exact names.

results = await self.graphiti.search(query, limit=limit)

ENTITY_FULLTEXT - Name Lookup

Direct fulltext search on entity nodes. When you say "Sarah", we search the name index. Fast and precise when you know who you're looking for.

results = await self.graphiti.search_entities(query, limit=limit)

STRUCTURAL - Graph Traversal

This is where the graph shines. Cypher queries that follow relationships:

MATCH (p:Person)-[r:WORKS_AT]->(o:Organization)
WHERE p.name =~ $name_pattern
  AND r.expired_at IS NULL
RETURN p.name as person, o.name as organization,
       r.title as title, r.created_at as created_at
ORDER BY r.created_at DESC

The Researcher includes predefined patterns for common queries: WORKS_AT, REPORTS_TO, BLOCKS, KNOWS, FRIEND_OF. But it can also write custom Cypher when the LLM determines the query needs something more complex.

One subtlety: Graphiti stores semantic relationships as RELATES_TO edges with a name property. So "knows" or "friend of" are filtered with:

WHERE r.name =~ '(?i).*(knows|friend).*'

TEMPORAL - Time-Filtered Search

Queries like "last week" or "yesterday" trigger temporal search. The Researcher parses time references and adds filters:

def _parse_time_reference(self, query: str) -> TimeRange | None:
    query_lower = query.lower()
    now = datetime.now(UTC)

    if "yesterday" in query_lower:
        start = (now - timedelta(days=1)).replace(hour=0, minute=0, second=0)
        end = start.replace(hour=23, minute=59, second=59)
        return TimeRange(start=start.timestamp(), end=end.timestamp())

    if "last week" in query_lower:
        start = now - timedelta(days=7)
        return TimeRange(start=start.timestamp(), end=now.timestamp())

    # ... handles "this week", "tomorrow", "next N days", etc.

For schedule queries (containing "calendar", "meeting", "schedule"), it queries CalendarEvent nodes by start_time instead of created_at - because you care when the meeting happens, not when you added it.

Handling Individual Failures

Parallel execution means individual strategies can fail without killing the whole search:

results = await asyncio.gather(*tasks, return_exceptions=True)

for label, result in zip(labels, results):
    if isinstance(result, Exception):
        logger.warning(f"[SWELL] Strategy {label} failed: {result}")
        results_by_technique[label] = []  # Empty, not fatal
    else:
        results_by_technique[label] = result

Neo4j down? Vector search still runs. API timeout on one query? The others complete. Resilience through parallelism.

Phase 3: Aggregation and Scoring

Multiple techniques return overlapping results. Aggregation handles this:

  1. Deduplication - By source_id and content prefix
  2. Strength-aware scoring - Results from STRUCTURAL queries may include relationship strengths (how close are these people? how important is this connection?)
  3. Composite ranking - Sort by combined score

The scoring formula boosts results with strong relationships:

def _calculate_result_score(self, result: RawSearchResult) -> float:
    base_score = result.vector_score if result.vector_score else 0.5

    if result.relationship_strengths:
        avg_strength = sum(result.relationship_strengths) / len(result.relationship_strengths)
        boost = 1 + (avg_strength * 0.3)  # configurable
    else:
        boost = 1.0

    return base_score * boost

A result with vector score 0.8 and relationship strength 0.9 scores 0.8 * (1 + 0.9 * 0.3) = 1.02. Relationship quality matters.

Phase 4: Report Synthesis

Raw results aren't user-friendly. Opus synthesizes them into a structured GraphIntelligenceReport:

class GraphIntelligenceReport(BaseModel):
    query: str
    direct_answer: str                    # What you asked for
    confidence: float                     # 0.0-1.0
    confidence_level: ConfidenceLevel     # HIGH/MEDIUM/LOW/UNCERTAIN
    evidence: list[EvidenceItem]          # Supporting facts with sources
    relationships: list[RelationshipDetail]  # Key connections
    key_entities: list[str]               # Primary entities mentioned
    as_of_date: str                        # Currency of report
    historical_notes: list[str]           # Notes about past state
    search_techniques_used: list[SearchTechnique]
    result_count: int
    related_queries: list[str]            # Suggested follow-ups
    gaps_identified: list[str]            # Missing information

The synthesis prompt tells Opus:

  • Lead with what the user wants to know
  • Never fabricate facts not in the results
  • Calibrate confidence based on source count, consistency, and recency
  • Include temporal context - when were these facts true?
  • Suggest follow-up queries
  • Identify gaps - what would improve the answer?

The critical constraint: "I don't have that in The Locker" is an acceptable answer. The Researcher never invents information.

Confidence Calculation

Confidence isn't just a number Opus picks. It's computed from evidence quality:

def _calculate_confidence(self, results: list[RawSearchResult]):
    # Factor 1: Number of sources (diminishing returns)
    source_factor = min(len(results) / 5, 1.0)

    # Factor 2: Average vector score
    scores = [r.vector_score for r in results if r.vector_score]
    avg_score = sum(scores) / len(scores) if scores else 0.5

    # Factor 3: Technique diversity
    techniques = len({r.source_technique for r in results})
    diversity = min(techniques / 3, 1.0)

    # Factor 4: Relationship strengths
    strengths = [s for r in results for s in r.relationship_strengths]
    avg_strength = sum(strengths) / len(strengths) if strengths else 0.5

    # Weighted combination
    confidence = (
        source_factor * 0.25 +
        avg_score * 0.30 +
        diversity * 0.20 +
        avg_strength * 0.25
    )

Multiple consistent sources from different techniques with strong relationships yield high confidence. A single weak vector match with no structural support yields low confidence. The math encodes epistemological intuition.

Entity References for Message Linking

One recent addition: key_entity_refs. When the Researcher finds entities, it extracts their UUIDs. This enables the system to create MENTIONED_IN relationships between entities and the messages that reference them.

class EntityReference(BaseModel):
    uuid: str           # Entity UUID from knowledge graph
    name: str           # Display name
    entity_type: str    # Person, Organization, etc.
    confidence: float   # How sure we are this is the right entity
    source_technique: str

Now when you ask "Who is Sarah?", the answer message gets linked to Sarah's node in the graph. Memory that remembers its own retrieval.

Configuration

The Researcher is configurable via YAML (hot-reloaded without restart):

# config/agents/researcher.yaml
model:
  planning: claude-opus-4-5-20251101
  synthesis: claude-opus-4-5-20251101
  temperature: 0.3

search:
  max_parallel_strategies: 4
  default_result_limit: 10
  max_total_results: 50
  timeout_seconds: 30

scoring:
  strength_boost_factor: 0.3

timeouts:
  planning: 15.0
  execution: 30.0
  synthesis: 20.0

Why Opus for both planning and synthesis? These are reasoning-heavy tasks. Analyzing query intent, selecting techniques, calibrating confidence - these require the reasoning depth that smaller models lack.

A Complete Example

Query: "What's blocking my project?"

Planning phase: Opus identifies this as a structural query about task dependencies. It generates strategies for BLOCKS and DEPENDS_ON relationships, plus vector search for semantic context about blockers.

Execution phase: Three parallel queries fan out. STRUCTURAL returns task chains from Neo4j. VECTOR finds facts mentioning blockers. Entity search finds relevant tasks by name.

Aggregation: Results are deduplicated. A task appearing in both STRUCTURAL and VECTOR results gets boosted. The blocker with highest combined score surfaces first.

Synthesis: Opus produces:

{
  "direct_answer": "Your project has 2 active blockers: waiting on

  "confidence_level": "high",

From ambiguous natural language to structured, actionable intelligence.

Why This Architecture?

Three principles drive the Researcher's design:

  1. AI-first planning - No keyword matching, no regex classification. The LLM understands your intent semantically and chooses techniques accordingly.
  2. Parallel execution - Multiple search techniques are complementary, not competing. Run them all, combine the results.
  3. Structured output - The GraphIntelligenceReport is channel-agnostic. CLI, Telegram, web - each can render it appropriately. The Researcher doesn't format for display; it structures for consumption.

The result is a retrieval system that reasons about how to find answers, not just what keywords to match. The Librarian doesn't just search the shelves - it understands what you're looking for.

The compass points true.
