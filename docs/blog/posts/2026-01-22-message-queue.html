Queueing for Resilience

The Orchestrator is thinking. Another message arrives. And another. What happens?

Before: messages waited in line at each channel's receive_message(). If the Orchestrator took too long, channels might timeout. Now: messages queue properly.

The Problem

Klabautermann handles messages from CLI, Telegram, and eventually Discord. Each channel calls the Orchestrator, which calls LLMs, which take time. During heavy usage, messages stack up.

Without a queue:

  • Channels block waiting for responses
  • Timeouts cause lost messages
  • No visibility into backlog
  • No control over overflow

The Queue

A new MessageQueue class handles the pressure:

from klabautermann.channels import MessageQueue, MessageQueueConfig

config = MessageQueueConfig(
    max_size=100,           # Messages before overflow
    overflow_policy="drop_oldest",  # What to do when full
    processing_timeout=300.0,       # 5 minutes per message
)

queue = MessageQueue(
    processor=orchestrator.handle_user_input,
    config=config,
)
await queue.start()

# Enqueue returns immediately
result = await queue.enqueue(thread_id, content, metadata)
print(f"Position: {result.queue_position}")

# Wait for response when ready
response = await queue.wait_for_result(result.item_id, timeout=60)

Overflow Policies

When the queue fills up, three options:

# Drop the oldest message (default)
overflow_policy="drop_oldest"

# Drop the new message trying to enter
overflow_policy="drop_newest"

# Reject new messages entirely
overflow_policy="reject"

Each has its use case. Drop oldest keeps new conversations responsive. Drop newest protects work already in progress. Reject gives hard backpressure to upstream systems.

Configuration

Environment variables for runtime tuning:

# Maximum queue depth
MESSAGE_QUEUE_MAX_SIZE=100

# What to do when full: drop_oldest, drop_newest, reject
MESSAGE_QUEUE_OVERFLOW=drop_oldest

# Processing timeout per message (seconds)
MESSAGE_QUEUE_TIMEOUT=300

FIFO Guarantee

Messages process in the order they arrive. First in, first out. Always. The test suite verifies this with concurrent message submission.
