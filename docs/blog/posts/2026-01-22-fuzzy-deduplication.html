The Duplicate Problem

Knowledge graphs have a dirty secret: duplicates creep in.

"I met John Doe at the conference." Later: "John D. sent me the report." Two nodes. One person. The graph fragments.

The Old Way

Simple exact matching:

WHERE toLower(p1.name) = toLower(p2.name)
   OR p1.email = p2.email

"John Doe" matches "john doe". But "John D." slips through. "Jon Doe" (typo) becomes a ghost entity.

The New Way

Fuzzy matching with confidence tiers:

from rapidfuzz import fuzz

similarity = fuzz.ratio("John Doe", "John D.") / 100.0
# Returns 0.727 - similar enough to flag

Now the Archivist sees the resemblance.

The Confidence Spectrum

Three tiers of duplicate handling:

  • >= 0.9: Auto-merge. Same email, same name. No human needed.
  • 0.7 - 0.9: Flag for review. Similar but uncertain. Human decides.
  • < 0.7: Ignore. Different enough to be different people.

The thresholds are configurable:

await archivist.detect_and_merge_duplicates(
    auto_merge_threshold=0.95,  # More conservative
    review_threshold=0.8,       # Higher bar for flagging
)

Detection Signals

Multiple signals combine into a similarity score:

  • Exact email match: 1.0 (definite duplicate)
  • Same email domain: 0.5 base + name similarity
  • Fuzzy name match: Direct ratio from rapidfuzz

Email matches trump name matches. "john@acme.com" in two places? Same John.

The Review Workflow

Medium-confidence matches get flagged:

MATCH (e1)-[r:POTENTIAL_DUPLICATE]->(e2)
SET r.similarity_score = 0.85,
    r.match_reasons = ["similar_name", "same_email_domain"],
    r.reviewed = false

A relationship in the graph, waiting for review.

Query flagged duplicates:

flagged = await archivist.get_flagged_duplicates()
# Returns: [DuplicateCandidate(name1="John Doe", name2="John D.", score=0.85)]

Resolve them:

# Merge: "Yes, same person"
await archivist.resolve_flagged_duplicate(uuid1, uuid2, merge=True)

# Dismiss: "No, different people"
await archivist.resolve_flagged_duplicate(uuid1, uuid2, merge=False)

Human judgment where it matters.

The Merge Operation

When entities merge:

  1. Transfer all incoming relationships to the keeper
  2. Transfer all outgoing relationships to the keeper
  3. Merge properties (keep existing, fill gaps from source)
  4. Delete the duplicate node

The graph heals. Knowledge consolidates.

The Architecture

detect_and_merge_duplicates()
├── process_duplicates()     # Main pipeline from deduplication.py
│   ├── find_duplicate_persons()
│   │   ├── Exact email matches (score=1.0)
│   │   ├── Domain matches + name similarity
│   │   └── Fuzzy name matches via rapidfuzz
│   ├── find_duplicate_organizations()
│   │   ├── Exact domain matches (score=1.0)
│   │   └── Fuzzy name matches
│   ├── Auto-merge high confidence (>= 0.9)
│   └── Flag medium confidence (0.7-0.9)

The Archivist calls process_duplicates() after each archival run. Duplicates surface. Merges happen. Flags get raised.
