No More Keywords

There's something satisfying about deleting code. Today we removed 50 lines of keyword-based intent classification and replaced them with... nothing. Well, almost nothing.

The Old Way

When Claude's API call failed, Klabautermann had a fallback. It would look at your message and match keywords:

# The old fallback logic
action_starts = ("send", "email", "schedule", "create", "draft", "book")
if any(text_lower.startswith(kw) for kw in action_starts):
    return IntentClassification(type=IntentType.ACTION, confidence=0.6)

ingest_starts = ("i met", "i talked", "i spoke", "i learned", "i just")
if any(text_lower.startswith(kw) for kw in ingest_starts):
    return IntentClassification(type=IntentType.INGESTION, confidence=0.6)

Simple. Predictable. And fundamentally at odds with what we're building.

The Problem with Keywords

Keywords work until they don't. "Send my regards to John" isn't an email action. "I just checked my calendar" isn't new information to store. And "What did I learn from Sarah?" is a search, not ingestion.

The AI-first principle in Klabautermann is clear: use pure LLM semantic understanding. No keyword matching. No regex patterns. If we're building a system that understands you, it should understand you - not pattern match.

The Issue

Issue #2 (AGT-P-001) was our P0 critical: "Remove keyword-based intent classification - use pure LLM." It had been sitting in the backlog since the gap analysis, flagged as violating the AI-first requirement.

The interesting part: the main classification path already used LLM. Claude Haiku analyzes your message and returns structured JSON:

{
    "intent_type": "search",
    "confidence": 0.9,
    "reasoning": "User is asking about a person they know",
    "extracted_query": "Who is Sarah?"
}

The keywords were only a fallback for when that LLM call failed. Network timeout? API error? Fall back to pattern matching.

The Fix

We could have made the fallback smarter. A smaller local model. More sophisticated heuristics. But that's still fighting the philosophy.

Instead, we chose graceful degradation:

except Exception as e:
    # AI-First: No keyword fallback
    # Return CONVERSATION with low confidence
    logger.warning(f"[SWELL] LLM classification failed: {e}")
    return IntentClassification(
        type=IntentType.CONVERSATION,
        confidence=0.3,  # Signals uncertainty
    )

When classification fails, we don't guess based on surface patterns. We return CONVERSATION with a low confidence score. The system knows it doesn't know. That's honest.

But Wait - V2 Doesn't Use This At All

Here's the twist: the v2 workflow (Think-Dispatch-Synthesize) doesn't use intent classification. It's all task planning now. Claude Opus analyzes your message and identifies all tasks that need to happen:

  • "I met Sarah from Acme. What's her email?" becomes:
    • Ingest task: "Sarah works at Acme"
    • Research task: "Find Sarah's email"

Multi-intent. Parallel execution. No classification needed.

So why fix the v1 fallback? Because it's still in the codebase. Because someone might enable it. Because having keyword matching anywhere is a statement about what kind of system this is. And this is an AI-first system.

The Spec Update

While we were at it, we updated specs/architecture/AGENTS.md. The config example still showed the old keyword lists:

intent_classification:
  search_keywords:
    - "who"
    - "what"
    - "when"

Now it reflects reality:

intent_classification:
  model: claude-3-5-haiku-20241022
  timeout: 5.0  # Seconds before graceful degradation

Specs should document what the system is, not what it used to be.

The Test

We added one new test:

async def test_llm_failure_returns_conversation(self, orchestrator):
    """When LLM fails, return CONVERSATION with low confidence."""
    with patch.object(orchestrator, "_call_classification_model") as mock:
        mock.side_effect = Exception("API timeout")
        intent = await orchestrator._classify_intent(
            "Send an email to John", None, "test-trace"
        )
        # Should NOT use keyword matching
        assert intent.type == IntentType.CONVERSATION
        assert intent.confidence == 0.3

Before this change, "Send an email to John" would have been classified as ACTION by the keyword fallback. Now it's CONVERSATION with low confidence - honest about its uncertainty.

Lines Changed

The diff is satisfying:

 4 files changed, 41 insertions(+), 75 deletions(-)

More deletions than insertions. Simpler. Cleaner. More aligned with what we're building.

The Bigger Picture

This was a P0 issue - critical priority. Not because the system was broken, but because it was philosophically inconsistent. An AI-first knowledge management system shouldn't have hardcoded keyword lists.

There's still one more AI-first violation in the backlog: AGT-P-018 - zoom level detection in the Researcher uses keyword matching. That's next.

One hardcoded list at a time, we're making this system what it claims to be.

The winds are fair.
