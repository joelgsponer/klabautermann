The Missing Pieces

Klabautermann could create calendar events. It could list them. But if you needed to reschedule that meeting? Delete that cancelled appointment? You were on your own.

Today we fixed that. Issues #209 and #210 - calendar update and delete. The final pieces of the CRUD puzzle.

The Gap

Looking at our GoogleWorkspaceBridge, the asymmetry was obvious:

# Email operations: full lifecycle
EMAIL_SEND, EMAIL_SEARCH, EMAIL_DELETE, EMAIL_ARCHIVE, EMAIL_LABEL

# Calendar operations: half a lifecycle
CALENDAR_CREATE, CALENDAR_LIST  # That's it?

You could tell Klabautermann to schedule a meeting. But when plans changed - as they always do - you had to open Google Calendar yourself. A PKM system that can't update your calendar isn't much of an assistant.

PATCH, Not PUT

When designing the update operation, we faced a choice. The Google Calendar API offers two methods:

  • events().update() - Full replacement. Send all fields or they disappear.
  • events().patch() - Partial update. Only change what you specify.

The answer was obvious once we thought about real usage. When you say "move the meeting to 3pm," you don't want to accidentally lose the description, the attendees, or the location. You just want to change the time.

async def update_event(
    self,
    event_id: str,
    title: str | None = None,
    start: datetime | None = None,
    end: datetime | None = None,
    description: str | None = None,
    location: str | None = None,
    attendees: list[str] | None = None,
    context: Any = None,
) -> UpdateEventResult:
    """
    Update an existing calendar event.

    Uses PATCH to allow partial updates - only specified fields are changed.
    """

Everything is optional except the event ID. Change the title? Pass a title. Leave it alone? Don't pass anything. The None values don't get included in the request body - they simply don't exist.

Building the Event Body

The implementation is clean precisely because we chose PATCH:

def do_update() -> dict[str, Any]:
    event_body: dict[str, Any] = {}

    if title is not None:
        event_body["summary"] = title
    if start is not None:
        event_body["start"] = {"dateTime": start.isoformat(), "timeZone": "UTC"}
    if end is not None:
        event_body["end"] = {"dateTime": end.isoformat(), "timeZone": "UTC"}
    if description is not None:
        event_body["description"] = description
    if location is not None:
        event_body["location"] = location
    if attendees is not None:
        event_body["attendees"] = [{"email": email} for email in attendees]

    event: dict[str, Any] = (
        self._calendar_service.events()
        .patch(calendarId="primary", eventId=event_id, body=event_body)
        .execute()
    )
    return event

Only populated fields make it into the body. Google handles the merge. No data loss.

Delete is Simple (and That's Good)

Calendar deletion is straightforward - which is exactly what you want for destructive operations:

async def delete_event(
    self,
    event_id: str,
    context: Any = None,
) -> DeleteEventResult:
    """Delete a calendar event."""
    await self.start()

    def do_delete() -> None:
        self._calendar_service.events().delete(
            calendarId="primary", eventId=event_id
        ).execute()

    try:
        await self._rate_limited_call("calendar", do_delete)
        logger.info(
            f"[BEACON] Calendar event deleted: {event_id}",
            extra={"event_id": event_id},
        )
        return DeleteEventResult(success=True, event_id=event_id)
    except (HttpError, ExternalServiceError) as e:
        logger.error(
            f"[STORM] Calendar event deletion failed: {e}",
            extra={"event_id": event_id},
        )
        return DeleteEventResult(success=False, event_id=event_id, error=str(e))

Event ID in, success or error out. The nautical logging continues: [BEACON] for success, [STORM] for failure.

Wiring the Executor

The Executor agent needed three pieces of plumbing:

1. Action parsing - recognizing the new action types:

elif action_type == "calendar_update":
    return ActionRequest(
        type=ActionType.CALENDAR_UPDATE,
        target=context.get("event_id"),
    )

elif action_type == "calendar_delete":
    return ActionRequest(
        type=ActionType.CALENDAR_DELETE,
        target=context.get("event_id"),
    )

2. Validation - making sure we have an event ID:

elif request.type == ActionType.CALENDAR_UPDATE:
    if not request.target:
        return ActionResult(
            success=False,
            message="I need the event ID to update. Which event should I modify?",
        )

3. Execution routing - calling the right handler:

elif request.type == ActionType.CALENDAR_UPDATE:
    return await self._handle_calendar_update(request, context, trace_id, ctx)

elif request.type == ActionType.CALENDAR_DELETE:
    return await self._handle_calendar_delete(request, context, trace_id, ctx)

The pattern matches exactly what we already had for emails and calendar creation. Consistency is its own documentation.

The Bigger Picture

CRUD is a pattern for a reason. Create, Read, Update, Delete - these are the four things you do with data. Having only two of them is like having a car with an accelerator but no brake.

With update and delete in place, Klabautermann can now manage your calendar properly. Not just add to it, but maintain it. Change things. Remove things. Act like the actual assistant it's meant to be.

The voyage continues.
