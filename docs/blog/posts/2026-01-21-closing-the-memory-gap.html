Closing the Memory Gap

Today we tackled two high-priority bugs that were undermining a core promise of Klabautermann: that it remembers what you tell it in a way that's actually useful later.

The Problem: Disconnected Memories

Imagine telling Klabautermann "I met John (john@example.com), PM at Acme." The Ingestor dutifully extracts John as a Person and Acme as an Organization. The knowledge graph grows. All seems well.

But then you ask: "What did I talk about with John?"

Silence. The Researcher finds John in the graph, but has no way to trace back to the specific message where you mentioned him. The entities were floating free, disconnected from their source.

This was Bug #350: the MENTIONED_IN relationships between entities and messages were never being created during Graphiti ingestion.

Two Information Flows, One Gap

The architecture had two separate paths for entity-message linking:

Flow 1 (working):
  Researcher → finds entities → _link_message_entities() → MENTIONED_IN ✓

Flow 2 (broken):
  Ingestor → Graphiti extracts entities → ??? → nothing linked ✗

The infrastructure existed - link_entities_to_message() was sitting there in message_linking.py. It just wasn't being called after Graphiti ingestion because add_episode() returned nothing useful.

The Fix: Close the Loop

We modified the flow to capture what Graphiti extracts and link it back:

  1. add_episode() now returns the episode name
  2. New method get_entities_from_episode() queries the graph for entities linked to that episode
  3. After ingestion, we call link_entities_to_message() with the extracted entities

Both the v1 (Orchestrator) and v2 (Ingestor agent) workflows now properly link entities to their source messages. The graph goes from disconnected islands to a proper web of relationships:

(Person {name: "John"})-[:MENTIONED_IN]->(Message)
(Organization {name: "Acme"})-[:MENTIONED_IN]->(Message)

The Second Bug: Truncated Calendars

Bug #349 was simpler but equally frustrating: asking "What's my schedule this week?" would show only 1-2 events even when you had 10+ across multiple calendars.

The culprit? A hardcoded LIMIT 10 in the schedule query. When you have events spread across Primary, Work, and Personal calendars, 10 is not enough. We bumped it to 100 - users expect to see their entire schedule, not a teaser.

// Before
MATCH (c:CalendarEvent)
WHERE c.start_time >= $start AND c.start_time <= $end
RETURN c
LIMIT 10  // This was the problem

// After
LIMIT 100  // Reasonable for a week's schedule

Testing the Fix

Both fixes pass the full test suite (1492 tests). The entity linking is tested through the existing message linking tests, and the schedule query change required only updating one test that was asserting on the exact parameter list.

What This Enables

With these fixes, Golden Scenario #1 from CLAUDE.md finally works end-to-end:

"I met John (john@example.com), PM at Acme"
→ Creates Person, Org, WORKS_AT
→ Links both to the source Message
→ "What did I talk about with John?" finds that exact message

This is the kind of contextual memory that makes Klabautermann useful as a true knowledge assistant rather than just a fancy note-taking app.

Lessons Learned

The entity linking bug had been lurking since the Graphiti integration was first implemented. The infrastructure was all there - someone had thoughtfully built message_linking.py - but the connection between ingestion and linking was never made.

It's a reminder that features aren't done until they're connected end-to-end. Building the parts is only half the work; wiring them together completes the circuit.

Fair winds ahead.
