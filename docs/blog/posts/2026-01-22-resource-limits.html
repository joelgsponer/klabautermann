Bounded Watchers

Monitoring tools that monitor everything except their own resource consumption. A classic irony.

Klabautermann's production stack has had resource limits from the start. The monitoring stack didn't. Until now.

The Problem

Docker containers are generous by default. No limits means a runaway process can consume all available memory, taking down everything else on the host. For the main application, we set boundaries early. The monitoring stack got overlooked.

Three services. Three potential memory hogs:

  • Prometheus: Stores metrics in memory and on disk. Long retention + many targets = high memory.
  • Grafana: Renders dashboards, runs queries. Complex dashboards = spiky memory usage.
  • Alertmanager: Routes alerts. Lightweight, but still unbounded.

The Allocation

Right-sizing based on typical workloads:

# Prometheus - the data store
deploy:
  resources:
    limits:
      cpus: "1.0"
      memory: 1G
    reservations:
      cpus: "0.25"
      memory: 256M

# Grafana - the dashboard renderer
deploy:
  resources:
    limits:
      cpus: "1.0"
      memory: 512M
    reservations:
      cpus: "0.25"
      memory: 128M

# Alertmanager - the router
deploy:
  resources:
    limits:
      cpus: "0.5"
      memory: 256M
    reservations:
      cpus: "0.1"
      memory: 64M

OOM Handling

When a container hits its memory limit, Docker kills it. What then?

restart: unless-stopped

The container restarts. For a monitoring service, this is acceptable. You lose a few seconds of metric collection. The world keeps turning.

For production apps, you'd want more sophisticated handling—graceful degradation, circuit breakers, alerts before OOM. But monitoring tools watching monitoring tools gets philosophical fast. Keep it simple: restart and move on.

Tuning Guidance

The defaults work for single-node setups scraping a handful of targets. Scale up? Adjust accordingly:

  • Many scrape targets: Increase Prometheus memory
  • Long retention: Increase Prometheus memory
  • Complex dashboards: Increase Grafana memory
  • Many concurrent users: Increase Grafana CPU and memory

Comments in the docker-compose file document these tradeoffs.
