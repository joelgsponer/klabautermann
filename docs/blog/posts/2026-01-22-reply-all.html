Reply All (Properly This Time)

Email threading seems simple until you try to implement it. Today we added the ability to reply to email threads, and discovered that "just reply to this email" involves more headers than you might expect.

The Problem

Klabautermann could already send emails via send_email(). But when you asked it to "reply to that email from Sarah," it would send a brand new email - unthreaded, lonely, floating in the inbox wilderness.

Gmail (and most email clients) groups messages into threads using specific headers. Without them, your reply looks like a new conversation entirely.

The Threading Headers

RFC 2822 defines how email threading works. Three headers matter:

  1. Message-ID: Every email has a unique identifier (like <abc123@mail.gmail.com>)
  2. In-Reply-To: Points to the Message-ID of the email you're replying to
  3. References: A chain of all Message-IDs in the thread

Gmail also uses its own threadId internally, which we need to include when sending via the API.

The Implementation

The new reply_to_email() method handles all of this:

async def reply_to_email(
    self,
    message_id: str,      # Original message to reply to
    body: str,            # Reply body
    reply_all: bool = False,
    draft_only: bool = False,
) -> SendEmailResult:

It fetches the original message to extract its headers, then constructs the reply with proper threading:

# Build References header: existing references + original message ID
references = original_references
if original_message_id:
    if references:
        references = f"{references} {original_message_id}"
    else:
        references = original_message_id

# Set threading headers
message["In-Reply-To"] = original_message_id
message["References"] = references

The Subject Line

A small but important detail: replies should have "Re:" prefixed to the subject (if not already present). We handle this without doubling up:

subject = original_subject
if not subject.lower().startswith("re:"):
    subject = f"Re: {subject}"

Reply All vs Reply

The reply_all parameter determines who gets the reply:

  • Reply: Just the original sender
  • Reply All: Original sender + everyone in To and CC

For reply-all, we extract the original recipients and add them to CC (filtering out the sender who goes in To):

if reply_all and (original_to or original_cc):
    all_recipients = []
    if original_to:
        all_recipients.extend([r.strip() for r in original_to.split(",") if r.strip()])
    if original_cc:
        all_recipients.extend([r.strip() for r in original_cc.split(",") if r.strip()])
    cc_recipients = ", ".join(r for r in all_recipients if r != original_sender)

Usage

With this feature, Klabautermann can now handle requests like:

"Reply to Sarah's email saying I'll attend the meeting"

The Executor routes this to reply_to_email(), which sends the reply in the same thread as Sarah's original message.

Testing

We added five unit tests covering the main scenarios:

  • Successful reply sending
  • Draft mode replies

  • Error handling for missing messages

All tests pass, bringing us to 1497 total tests in the suite.
