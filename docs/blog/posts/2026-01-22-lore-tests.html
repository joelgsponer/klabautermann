Saga Continuation (#122)

Chapter numbers matter. Chapter 1 starts a saga. Chapter 2 continues it. If chapters don't increment correctly, the story breaks.

async def test_continue_saga_increments_chapter(self, ...):
    """Continuing a saga should increment the chapter number."""
    saga = ActiveSaga(
        saga_id="saga-continuation-test",
        saga_name="The Tale of Testing",
        last_chapter=2,
        last_told=now_ms - 1000,
    )

    content, chapter = await bard._continue_saga(saga, trace_id="test-122")

    assert chapter == 3

Simple. Last chapter was 2, new chapter should be 3.

Cross-Channel Persistence (#123)

The Captain moves between CLI and Telegram. The saga follows. Test this by starting on CLI, continuing on Telegram:

async def test_same_saga_across_channels(self, ...):
    """Same saga should be accessible from different channels."""
    # Tell on CLI (chapter 1)
    _, saga_id, _ = await bard.start_new_saga(channel="cli", ...)

    # Continue on Telegram (chapter 2)
    saga = ActiveSaga(saga_id=saga_id, last_chapter=1, ...)
    _, chapter = await bard._continue_saga(saga, channel="telegram", ...)

    assert chapter == 2

    # Query shows both channels
    cross_channel = await bard.get_cross_channel_story(saga_id=saga_id, ...)
    assert cross_channel[0]["channel"] == "cli"
    assert cross_channel[1]["channel"] == "telegram"

One saga. Two channels. Both tracked.

The Golden Scenario (#124)

Section 8.2 of LORE_SYSTEM.md describes the canonical cross-conversation flow:

  1. CLI: User asks about tasks. Bard starts saga, Chapter 1.
  2. Telegram: User checks calendar. Bard continues with Chapter 2.
  3. CLI (next day): User asks "What story were you telling?" Bard retrieves saga, continues with Chapter 3.
@pytest.mark.e2e
class TestLoreGoldenScenarioCrossConversation:
    """
    Golden Scenario: Cross-Conversation Saga (#124)

    Reference: specs/architecture/LORE_SYSTEM.md Section 8.2
    """

    async def test_complete_cross_conversation_flow(self, ...):
        # Step 1: CLI start
        _, saga_id, ch1 = await bard.start_new_saga(channel="cli", ...)
        assert ch1 == 1

        # Step 2: Telegram continue
        result2 = await bard.salt_response(..., channel="telegram", ...)
        assert result2.chapter == 2

        # Step 3: CLI retrieve and continue
        result3 = await bard.salt_response(..., channel="cli", ...)
        assert result3.chapter == 3

        # Verify cross-channel journey
        cross_channel = await bard.get_cross_channel_story(saga_id=saga_id, ...)
        assert cross_channel[0]["channel"] == "cli"      # Chapter 1
        assert cross_channel[1]["channel"] == "telegram" # Chapter 2
        assert cross_channel[2]["channel"] == "cli"      # Chapter 3

The spec becomes the test. The test proves the code.

Edge Cases

Two more scenarios round out the E2E tests:

async def test_saga_completes_at_max_chapters(self, ...):

Storm mode: Does the Bard stay silent when urgency matters?

async def test_storm_mode_skips_tidbit(self, ...):
    """Storm mode should prevent tidbit addition even with active saga."""
    result = await bard.salt_response(
        "URGENT: Server is down!",
        storm_mode=True,
        ...
    )

    assert result.tidbit_added is False
    assert result.storm_mode_skipped is True
    assert result.salted_response == "URGENT: Server is down!"

No tales during emergencies. The Bard knows when to be quiet.

Mock Patterns

Testing async Neo4j calls requires careful mock management. The pattern that emerged:

# Set up expectations for multiple calls
mock_neo4j.execute_query.side_effect = [
    [],                    # First call: no active sagas
    [{"uuid": "ep-1"}],    # Second call: save episode
]

await bard.start_new_saga(...)

# Reset before next operation
mock_neo4j.execute_query.reset_mock()
mock_neo4j.execute_query.side_effect = None
mock_neo4j.execute_query.return_value = [{"uuid": "ep-2"}]

await bard._continue_saga(...)

The reset_mock() call clears the side_effect list so subsequent operations don't exhaust it. Without this, StopAsyncIteration errors crash the tests.
