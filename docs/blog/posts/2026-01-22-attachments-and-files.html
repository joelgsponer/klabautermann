The Attachment Problem

Email attachments are ubiquitous. PDFs, spreadsheets, images - they arrive daily. And yet, in a system built to understand "save that report Sarah sent," we couldn't actually... save it. That changed today.

The Model

First, we needed to represent what an attachment is:

class EmailAttachment(BaseModel):
    """Email attachment metadata from Gmail."""

    attachment_id: str  # Gmail's internal attachment ID for downloading
    filename: str
    mime_type: str
    size: int  # Size in bytes

    @property
    def size_human(self) -> str:
        """Human-readable file size."""
        if self.size < 1024:
            return f"{self.size} B"
        elif self.size < 1024 * 1024:
            return f"{self.size / 1024:.1f} KB"
        else:
            return f"{self.size / (1024 * 1024):.1f} MB"

The attachment_id is Gmail's reference for the attachment data - you can't get the bytes without it. The size_human property is a convenience - "5.2 MB" reads better than "5452595 bytes."

Parsing Multipart

Email isn't simple text. Modern emails are MIME multipart messages: a text body, maybe an HTML version, and zero or more attachments. The structure can be deeply nested:

multipart/mixed
├── multipart/alternative
│   ├── text/plain (body)
│   └── text/html (formatted body)
├── application/pdf (attachment)
└── image/png (attachment)

We needed a recursive parser. The old code just grabbed the first text/plain part. The new code walks the entire tree:

def _extract_parts(
    self, parts: list[dict[str, Any]]
) -> tuple[str | None, list[EmailAttachment]]:
    """
    Recursively extract body text and attachments from email parts.
    """
    body = None
    attachments: list[EmailAttachment] = []

    for part in parts:
        mime_type = part.get("mimeType", "")
        filename = part.get("filename", "")
        part_body = part.get("body", {})

        # Check for nested multipart
        if "parts" in part:
            nested_body, nested_attachments = self._extract_parts(part["parts"])
            if nested_body and not body:
                body = nested_body
            attachments.extend(nested_attachments)

        # Check for text body
        elif mime_type == "text/plain" and part_body.get("data") and not filename:
            if not body:  # Only take first text/plain
                body = base64.urlsafe_b64decode(part_body["data"]).decode("utf-8")

        # Check for attachments (has filename and attachmentId)
        elif filename and part_body.get("attachmentId"):
            attachment = EmailAttachment(
                attachment_id=part_body["attachmentId"],
                filename=filename,
                mime_type=mime_type,
                size=part_body.get("size", 0),
            )
            attachments.append(attachment)

    return body, attachments

The key insight: attachments have both a filename and an attachmentId. Inline images might have an ID but no filename. Empty text parts might have neither. The combination is what identifies a real attachment.

Downloading Bytes

Gmail doesn't include attachment data in the message response - too expensive. You request it separately:

async def download_attachment(
    self,
    message_id: str,
    attachment_id: str,
    context: Any = None,
) -> bytes:
    """Download an email attachment's raw bytes."""
    await self.start()

    def do_download() -> bytes:
        attachment = (
            self._gmail_service.users()
            .messages()
            .attachments()
            .get(userId="me", messageId=message_id, id=attachment_id)
            .execute()
        )
        data = attachment.get("data", "")
        return base64.urlsafe_b64decode(data)

    return await self._rate_limited_call("gmail", do_download)

The attachment data comes base64-encoded, same as inline body content. Decode and you have raw bytes.

Saving to Disk

Downloading is only half the story. Users want to save files. This brought a wrinkle: duplicate filenames.

async def save_attachment(
    self,
    message_id: str,
    attachment: EmailAttachment,
    save_path: str,
    context: Any = None,
) -> str:
    """Download and save an attachment to the local filesystem."""
    from pathlib import Path as FilePath

    data = await self.download_attachment(message_id, attachment.attachment_id)

    save_dir = FilePath(save_path)
    save_dir.mkdir(parents=True, exist_ok=True)

    file_path = save_dir / attachment.filename

    # Handle duplicate filenames
    if file_path.exists():
        base = FilePath(attachment.filename).stem
        ext = FilePath(attachment.filename).suffix
        counter = 1
        while file_path.exists():
            file_path = save_dir / f"{base}_{counter}{ext}"
            counter += 1

    file_path.write_bytes(data)
    return str(file_path)

If report.pdf exists, save as report_1.pdf. If that exists, report_2.pdf. Simple, predictable.

The Filesystem Bridge

Saving attachments is one thing. But what if the agent needs broader file access? Reading configuration, writing logs, moving files around?

Enter the FilesystemBridge. It wraps the MCP filesystem server - the same protocol that powers Claude's file access - for sandboxed operations:

@dataclass
class FilesystemConfig:
    """Configuration for filesystem MCP server."""

    allowed_paths: list[str] = field(default_factory=list)
    timeout: float = 30.0

    def __post_init__(self) -> None:
        """Validate and resolve allowed paths."""
        if not self.allowed_paths:
            self.allowed_paths = [str(Path.cwd())]
        # Resolve all paths to absolute
        self.allowed_paths = [str(Path(p).resolve()) for p in self.allowed_paths]

The allowed_paths list is the security boundary. Try to read /etc/passwd when only /app/data is allowed? Rejected. The MCP server handles this at the protocol level, and we double-check locally for binary operations.

Wiring It Up

The bridge spawns the MCP server as a subprocess:

async def start(self) -> None:
    """Start the filesystem MCP server."""
    if self._started:
        return

    command = [
        "npx",
        "-y",
        "@modelcontextprotocol/server-filesystem",
        *self.config.allowed_paths,
    ]

    server_config = MCPServerConfig(
        name=self.SERVER_NAME,
        command=command,
        timeout=self.config.timeout,
    )

    await self._client.start_server(server_config)
    self._started = True

The allowed paths become command-line arguments to the server. Everything else flows through JSON-RPC over stdio.

Operations

The bridge exposes the standard file operations:

  • read_file() / read_file_bytes() - Read text or binary files
  • write_file() / write_file_bytes() - Write with auto-directory creation
  • list_directory() - Get directory contents
  • create_directory() - Create directories recursively
  • move_file() - Rename or move files
  • get_file_info() - Get size, modified time, etc.

Each operation validates paths against the allowed list before executing. Defense in depth.
