Once Is Not Enough

Issue #211 sat there, quietly judging us. "Schedule daily standup at 9am." A perfectly reasonable request. And yet Klabautermann's response would be to create a single event. Tomorrow? Create another. Next week? Five more commands.

That's not how calendars work. That's not how humans think. It was time to fix it.

The Standard

RFC 5545 defines how calendar applications handle recurrence. It's the iCalendar specification, and at its heart is the RRULE - the recurrence rule. Google Calendar, Apple Calendar, Outlook - they all speak this language:

RRULE:FREQ=DAILY
RRULE:FREQ=WEEKLY;BYDAY=MO,WE,FR
RRULE:FREQ=MONTHLY;BYMONTHDAY=15
RRULE:FREQ=YEARLY;COUNT=10

The format is dense but logical. Frequency, day constraints, count limits, end dates. Everything you need to express "every Tuesday and Thursday until June" in a single string.

The Builder Pattern

Nobody wants to construct RRULE strings by hand. We created RecurrenceBuilder - a helper class that speaks human:

class RecurrenceBuilder:
    @staticmethod
    def daily(count: int | None = None, until: datetime | None = None) -> str:
        rule = "RRULE:FREQ=DAILY"
        if count:
            rule += f";COUNT={count}"
        elif until:
            rule += f";UNTIL={until.strftime('%Y%m%dT%H%M%SZ')}"
        return rule

    @staticmethod
    def weekly(
        days: list[str] | None = None,
        count: int | None = None,
        until: datetime | None = None
    ) -> str:
        rule = "RRULE:FREQ=WEEKLY"
        if days:
            rule += f";BYDAY={','.join(days)}"
        if count:
            rule += f";COUNT={count}"
        elif until:
            rule += f";UNTIL={until.strftime('%Y%m%dT%H%M%SZ')}"
        return rule

The pattern continues for monthly(), yearly(), and the convenience method everyone actually wants:

@staticmethod
def weekdays(count: int | None = None, until: datetime | None = None) -> str:
    """Every weekday (Monday through Friday)."""
    return RecurrenceBuilder.weekly(["MO", "TU", "WE", "TH", "FR"], count, until)

Now "schedule daily standup at 9am" translates to RecurrenceBuilder.weekdays(). One line.

The model needed two new fields:

What's Different Now

Before:

"Schedule daily standup at 9am for the next month" → Creates one event. You're on your own for the other 29.

After:

"Schedule daily standup at 9am for the next month" → Creates a recurring event with RRULE:FREQ=DAILY;COUNT=30. All 30 instances appear on your calendar.

Before:

"Add weekly team meeting every Tuesday at 2pm" → Creates one meeting on one Tuesday.

After:

"Add weekly team meeting every Tuesday at 2pm" → Creates a recurring series with RRULE:FREQ=WEEKLY;BYDAY=TU. Every Tuesday, forever, until you say stop.

The RFC Is Your Friend

RFC 5545 is one of those standards that got it right. It's been stable since 2009 because it captured the right abstractions. Daily, weekly, monthly, yearly. Count limits and end dates. Specific days of the week or month.

We didn't need to invent anything. We just needed to expose what was already there, hiding the complexity behind a builder that speaks human.

RecurrenceBuilder.weekly(["TU", "TH"]) is easier to write than RRULE:FREQ=WEEKLY;BYDAY=TU,TH. But the latter is what Google Calendar understands, what Apple Calendar understands, what every RFC 5545-compliant system understands.

Klabautermann now speaks that language.
