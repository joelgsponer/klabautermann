Trust But Verify

Skills are configuration. Configuration has bugs. And configuration bugs are the worst kind - they fail silently at runtime, often in production, when the user types something you didn't test. Today we added a validation layer that catches these before they cause trouble.

The Problem

Consider a skill with a typo:

---
name: lookup-Person  # Uppercase P - invalid
description: Find people
klabautermann-task-type: research
klabautermann-agent: Researcher  # Should be lowercase
---

This would load fine. The orchestrator would accept it. But it would never match any user requests because the AI expects lowercase identifiers. And debugging "why doesn't my skill work" when the answer is "capital letter in line 2" is nobody's idea of fun.

The Validator

The SkillValidator class checks for common issues:

class SkillValidator:
    """
    Validates skill definitions against schema and dependency rules.

    Checks:
    - Name format (lowercase, hyphens, max 64 chars)
    - Required fields (name, description)
    - Description quality (min length, contains trigger phrases)
    - Tool references (warns if unknown)
    - Agent references (warns if unknown)
    - Orchestrator config completeness
    - Payload schema validity
    """

Validation Rules

Name Format

# Valid name pattern: lowercase letters, numbers, and hyphens
NAME_PATTERN = re.compile(r"^[a-z][a-z0-9-]*$")
MAX_NAME_LENGTH = 64

def _validate_name(self, skill, result):
    name = skill.name

    if len(name) > MAX_NAME_LENGTH:
        result.add_error("name", f"Name too long ({len(name)} chars, max {MAX_NAME_LENGTH})")

    if not NAME_PATTERN.match(name):
        result.add_error("name",
            "Name must be lowercase, start with a letter, and contain only "
            "letters, numbers, and hyphens")

Names must be lowercase, start with a letter, and contain only letters, numbers, and hyphens. This matches Claude Code's skill naming conventions.

Description Quality

def _validate_description(self, skill, result):
    description = skill.description

    # Check minimum length
    if len(description) < 10:
        result.add_error("description",
            "Description too short (min 10 chars) - describe what the skill does")

    # Warn if no trigger phrases
    trigger_indicators = ["when", "use when", "use for", "triggers on", "say"]
    has_trigger = any(indicator in description.lower() for indicator in trigger_indicators)
    if not has_trigger:
        result.add_warning("description",
            "Description should include trigger phrases (e.g., 'Use when user asks...')")

The description is what the AI uses to match user requests to skills. A description without trigger phrases ("Use when...") makes discovery unreliable.

Orchestrator Config Consistency

def _validate_orchestrator_config(self, skill, result):
    config = skill.klabautermann

    # If task_type is set, agent should be set too
    if config.task_type and not config.agent:
        result.add_error("klabautermann-agent",
            f"klabautermann-task-type is '{config.task_type}' but klabautermann-agent is missing")

    # Validate task_type and agent consistency
    if config.task_type and config.agent:
        expected_agents = {
            "ingest": "ingestor",
            "research": "researcher",
            "execute": "executor",
        }
        expected = expected_agents.get(config.task_type)
        if expected and config.agent != expected:
            result.add_warning("klabautermann-agent",
                f"Task type '{config.task_type}' typically uses agent '{expected}', not '{config.agent}'")

If you set task-type: research, you probably meant agent: researcher. Using a different agent isn't wrong, but it's unusual enough to warrant a warning.

Errors vs Warnings

The validation distinguishes between errors (will definitely cause problems) and warnings (might be intentional):

@dataclass
class ValidationError:
    """A single validation error."""
    field: str
    message: str
    severity: str = "error"  # "error" or "warning"

@dataclass
class ValidationResult:
    """Result of validating a skill definition."""
    skill_name: str
    skill_path: Path | None = None
    errors: list[ValidationError] = field(default_factory=list)
    warnings: list[ValidationError] = field(default_factory=list)

    @property
    def is_valid(self) -> bool:
        """Whether the skill passed validation (no errors, warnings OK)."""
        return len(self.errors) == 0

Unknown tools or models generate warnings - they might be new tools we haven't added to the known list. Invalid names or missing required fields generate errors.

Strict Mode

For CI/CD pipelines, there's strict mode:

# Strict: warnings become errors
result = validate_skill(skill, strict=True)

if not result.is_valid:
    print(result.format_report())
    sys.exit(1)

This is useful in pre-commit hooks or automated testing where you want zero tolerance for issues.

Human-Readable Reports

def format_report(self) -> str:
    """Format validation result as a human-readable report."""
    lines = [f"Validation result for skill: {self.skill_name}"]
    if self.skill_path:
        lines.append(f"Path: {self.skill_path}")
    lines.append("")

    if self.is_valid and not self.has_warnings:
        lines.append("OK - No issues found")
    else:
        if self.errors:
            lines.append(f"Errors ({len(self.errors)}):")
            for error in self.errors:
                lines.append(f"  - {error}")
        if self.warnings:
            lines.append(f"Warnings ({len(self.warnings)}):")
            for warning in self.warnings:
                lines.append(f"  - {warning}")

    return "\n".join(lines)

Convenience Functions

# Validate a loaded skill
result = validate_skill(skill)

# Validate a file directly (parses and validates)
result = validate_skill_file(Path(".claude/skills/my-skill/SKILL.md"))

# Validate all skills in a loader
results = validate_all_skills(loader)
for name, result in results.items():
    if not result.is_valid:
        print(result.format_report())

The validate_all_skills() function is useful for auditing an entire skills directory.

Custom Validators

For projects with custom tools or agents:

validator = SkillValidator(
    known_tools=frozenset({"Read", "Grep", "CustomProjectTool"}),
    known_agents=frozenset({"researcher", "executor", "custom_agent"}),
)

This suppresses warnings for tools and agents that exist in your project but aren't in the default known sets.
