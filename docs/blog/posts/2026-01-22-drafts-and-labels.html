Save for Later

Email workflows rarely end with "send." Sometimes you need to draft a response, sleep on it, edit it the next morning. Sometimes you need to organize hundreds of emails into a folder structure that makes sense. Today Klabautermann learned both.

The Draft Model

Gmail's draft API returns more than just message content. There's a draft ID (different from the message ID), thread association, and all the headers you'd expect. The model captures this:

class EmailDraft(BaseModel):
    """Email draft from Gmail."""

    id: str  # Draft ID (different from message ID)
    message_id: str  # Associated message ID
    thread_id: str | None = None
    subject: str
    to: str | None = None
    cc: str | None = None
    body: str | None = None
    snippet: str = ""

The distinction between id and message_id matters. The draft ID is what you use to update or send the draft. The message ID is how Gmail tracks the underlying content. Confuse them and your updates silently fail.

Listing Drafts

Gmail's drafts.list returns minimal information - just IDs. To get useful details, you need a second request for each draft:

async def list_drafts(
    self,
    max_results: int = 20,
    context: Any = None,
) -> list[EmailDraft]:
    """List email drafts."""
    await self.start()

    def do_list_drafts() -> list[dict[str, Any]]:
        results = (
            self._gmail_service.users()
            .drafts()
            .list(userId="me", maxResults=max_results)
            .execute()
        )
        return results.get("drafts", [])

    def get_draft_details(draft_id: str) -> dict[str, Any]:
        return (
            self._gmail_service.users()
            .drafts()
            .get(userId="me", id=draft_id, format="full")
            .execute()
        )

    raw_drafts = await self._rate_limited_call("gmail", do_list_drafts)
    drafts: list[EmailDraft] = []

    for raw_draft in raw_drafts:
        draft_id = raw_draft.get("id", "")
        full_draft = await self._rate_limited_call(
            "gmail",
            lambda d=draft_id: get_draft_details(d),
        )
        # Parse headers and body...
        drafts.append(EmailDraft(...))

Each draft requires its own API call. For large draft collections, this could get expensive. But users rarely have hundreds of drafts, and the rate limiter keeps us in good standing.

Updating Drafts

The update operation has a subtle requirement: preserve what you don't change. If you only want to update the subject, the existing recipients should remain:

async def update_draft(
    self,
    draft_id: str,
    to: str | None = None,
    subject: str | None = None,
    body: str | None = None,
    cc: str | None = None,
    context: Any = None,
) -> DraftOperationResult:
    """Update an existing draft."""
    await self.start()

    # First get the existing draft to preserve fields
    existing = await self.get_draft(draft_id)
    if not existing:
        return DraftOperationResult(
            success=False,
            draft_id=draft_id,
            operation="update",
            error="Draft not found",
        )

    # Use existing values if not provided
    final_to = to if to is not None else existing.to
    final_subject = subject if subject is not None else existing.subject
    final_body = body if body is not None else existing.body or ""
    final_cc = cc if cc is not None else existing.cc

Sending Drafts

Once you're ready, sending is straightforward:

async def send_draft(
    self,
    draft_id: str,
    context: Any = None,
) -> DraftOperationResult:
    """Send an existing draft."""
    await self.start()

    def do_send() -> dict[str, Any]:
        return (
            self._gmail_service.users()
            .drafts()
            .send(userId="me", body={"id": draft_id})
            .execute()
        )

    result = await self._rate_limited_call("gmail", do_send)
    return DraftOperationResult(
        success=True,
        draft_id=draft_id,
        message_id=result.get("id"),
        operation="send",
    )

The draft disappears after sending, replaced by a real message in your Sent folder.

Custom Labels

Gmail labels are more flexible than folders. A message can have multiple labels. Labels can be nested using "/" separators. They can be hidden from the UI or shown only when they have unread messages.

async def create_label(
    self,
    name: str,
    label_list_visibility: str = "labelShow",
    message_list_visibility: str = "show",
    context: Any = None,
) -> GmailLabel:
    """Create a custom Gmail label."""
    await self.start()

    def do_create_label() -> dict[str, Any]:
        label_body = {
            "name": name,
            "labelListVisibility": label_list_visibility,
            "messageListVisibility": message_list_visibility,
        }
        return (
            self._gmail_service.users()
            .labels()
            .create(userId="me", body=label_body)
            .execute()
        )

    result = await self._rate_limited_call("gmail", do_create_label)
    return GmailLabel(
        id=result.get("id", ""),
        name=result.get("name", name),
        type="user",
    )

The visibility settings control where labels appear:

  • labelShow - Always visible in label list
  • labelShowIfUnread - Only visible when there are unread messages
  • labelHide - Hidden from label list (but still usable)

Nested Labels

Creating "Projects/Work" gives you a nested structure in Gmail's UI. The API doesn't treat this specially - it's just a label name containing a slash. But the UI renders it as a hierarchy:

Projects
└── Work
    └── Urgent

Create "Projects", "Projects/Work", and "Projects/Work/Urgent" separately. Gmail doesn't auto-create parent labels.

Updating Labels

Labels can be renamed or have their visibility changed:

async def update_label(
    self,
    label_id: str,
    name: str | None = None,
    label_list_visibility: str | None = None,
    message_list_visibility: str | None = None,
    context: Any = None,
) -> GmailLabel:
    """Update a custom Gmail label."""
    await self.start()

    def do_update_label() -> dict[str, Any]:
        label_body: dict[str, Any] = {}
        if name is not None:
            label_body["name"] = name
        if label_list_visibility is not None:
            label_body["labelListVisibility"] = label_list_visibility
        if message_list_visibility is not None:
            label_body["messageListVisibility"] = message_list_visibility

        return (
            self._gmail_service.users()
            .labels()
            .patch(userId="me", id=label_id, body=label_body)
            .execute()
        )

Note the use of patch() rather than update(). The patch method only sends fields you want to change, leaving others untouched.
